①摘要与设计哲学

本项目从"一个旋转的立方体"起步,迭代出一款功能完整的 3D 赛车游戏,随后以同一套工程底座、用多智能体并行扩展出 34 款不同类型的游戏,覆盖竞速、跑酷、节奏、射击、塔防、物理、益智、平台、迷宫、体素,直到摄像头/语音等硬件交互。

贯穿始终的四条设计原则:

  • 单文件、零构建:每款游戏是一个自包含 HTML,双击即玩,无打包、无后端。
  • 离线优先:除按需的 CDN 库(Three.js / matter.js / MediaPipe)与字体外不依赖外部素材;贴图、音效、配乐全部用代码现场生成。
  • 统一美学:合成波(OutRun)霓虹风格——日落渐变、扫描线、霓虹边框、像素字,形成跨 34 款的视觉一致性。
  • 可演示:每款内建 ?demo 自动演示 AI 与 ?record 自录,使"无人值守生成演示视频"成为流水线。
核心洞察:不同"游戏类型"的差异,主要不在底层引擎,而在四个旋钮——维度(2D/3D/第一人称)、是否需要物理、输入方式、胜负结构。底座不变,换旋钮即得新类型。这正是 34 款能共享同一工程内核的根本原因。

②通用底座:任何游戏的 80%

每款游戏都由同样的 6 块构成,这套"内核"占了任意游戏代码量的约 80%:

积木实现要点
游戏循环requestAnimationFrame + delta time(dt = (now-last)/1000,并 clamp(dt,0.05) 防卡顿跳变);所有运动按 ×dt 推进,保证不同帧率手感一致。
状态机menu → playing → over/win → 重开 的闭环;遮罩显隐与输入门控由状态驱动。后期统一加 try/catch 包裹 update/draw,使单帧异常不打断 rAF 链。
碰撞2D 用圆-圆 / 圆-矩形 / AABB;3D 用 THREE.Box3.intersectsBox;高速场景用跨平面扫描(记录 prevZ,跨过判定面才触发)防穿透漏判。
对象池 / 程序化生成障碍/金币/敌人/路面分段循环复用(飞出屏幕回收再随机),避免无限增长导致卡顿。
反馈(juice)命中/得分/失败的视觉+听觉反馈:撞击顿帧、红闪、镜头抖动、粒子、连击弹字。
存档 / 封装localStorage 最高分、PWA 离线、国际化、双镜头/分屏;读写一律 try/catch(隐私模式下 localStorage 会抛异常)。

伪 3D 的两个招牌技巧(赛车/跑酷)

赛车与跑酷并非真做曲面赛道,而是用两个函数把"直路"伪装成弯道与坡道:

  • bendX(z) — 弯道:离镜头越远的物体横向偏移越大(偏移 ∝ 曲率 × 距离²),路面/护栏/车辆一起套用,直路看起来就蜿蜒了;近处偏移≈0,所以碰撞仍按车道判定,公平不变。
  • hillY(z) — 坡道:用一整块带网格细分的地面平面,每帧按"赛道位置"位移它的顶点(positions.setZ(...)),单网格、无接缝;车辆与地面用同一个 hillY 函数,所以车永远精准贴坡。

这类"伪 3D"以极小代价获得了立体观感,是 90 年代街机赛车的经典做法在 WebGL 上的复刻。

着色器隧道:把游戏画进 fragment shader

霓虹隧道用一个铺满屏的四边形 + 自定义片元着色器,随 uTime 推进,用极坐标与 raymarching 思路实时生成扭曲发光的隧道壁。这是与其余 33 款最不同的渲染范式——没有"物体",只有一段每像素求值的数学。审查中修复了它最隐蔽的 bug:高速时单帧推进量超过碰撞判定窗口导致穿墙漏判,改为跨判定面触发。

④音频技术

所有音效与配乐不用任何音频文件,全部用 Web Audio API 实时合成:

  • 音效:振荡器(sine/square/sawtooth/triangle)+ 增益包络(快速 attack、指数 decay)合成币声、击中、爆炸、UI 提示。
  • 配乐:旋律 + 贝斯 + 鼓点的循环,用前瞻调度器(以 audioCtx.currentTime 提前排定音符,setInterval 仅做调度)保证采样级精准、无卡顿。
  • 空间音频(赛车):对手车擦身时用 StereoPannerNode 按左右方位播"嗖"声。
  • 程序化引擎声(赛车):一条持续锯齿波,频率随车速实时升降。
  • 节奏游戏音画同步(节奏/跳舞):谱面音符的位置由 音符时间 − audioCtx.currentTime 推导,判定窗口(Perfect/Good/Miss)以采样时钟对齐,而非依赖 rAF 时间。
  • 语音合成:赛车的领航员、预告片旁白早期用 SpeechSynthesis(后改为系统 TTS 离线生成音频)。
浏览器自动播放策略:AudioContext 必须在用户手势(点击开始)之后创建/resume(),否则被挂起。34 款一律遵守;演示模式下虽免手势创建(用于纯画面录制)也以 try/catch 容错、不报错。

⑦录制与视频生产管线

把"会跑的游戏"变成"可分享的预告片",建立了一条完全自动化的流水线:

  1. 实机录制:
    • canvas 类(28 款)— canvas.captureStream(30) + MediaRecorder,录主画布并下载 webm;
    • DOM/SVG 类(2 款,无 canvas 无法 captureStream)— 改用 ffmpeg gdigrab 全屏屏幕录制。
  2. 素材整形:游戏多为竖屏/方屏,ffmpeg 用模糊填充(同一画面 split → 一路放大裁切高斯模糊作背景、一路等比缩放作前景 overlay 居中)铺满 1080×1920 竖屏,既看全又无黑边。
  3. 画面渲染:用 HeyGen 开源的 HyperFrames(HTML→MP4)把"冷开场→镀铬标题→实机片段→CTA"的合成波模板逐帧渲染。关键约束:GSAP 时间轴必须暂停且可 seek(渲染器逐帧 seek 截图),禁用 Date.now/Math.random/setTimeout 保证确定性。
  4. 旁白:Windows 系统 TTS(System.Speech + 中文音色 Huihui)用 SSML prosody 压低音调离线生成讲解音频。
  5. 配乐:用 Node 脚本合成 PCM 写出 WAV(锯齿波旋律 + 三角波贝斯 + 合成鼓)。
  6. 合成:ffmpeg 把画面 + 旁白(adelay 定时)+ 配乐(降音量)amix 混音并 alimiter 限幅,输出最终 MP4。
关键限制:HyperFrames 逐帧渲染无法跑实时 3D 游戏、也录不到网页声音——所以实机画面只能先录成视频再嵌入合成,而非让游戏在渲染器里实时运行。这是整条管线最核心的取舍。

⑧持久化 / PWA / 国际化

  • localStorage:最高分、最快用时、涂装选择、语言、幽灵竞速轨迹(序列化的位置序列)。
  • PWA(赛车):manifest.json + Service Worker 缓存外壳与 CDN,可装到桌面、离线运行——但 Service Worker 不支持 file://,需经本地 http 服务。
  • 国际化(赛车):中/英/日三语,词典 + tr(key) 函数 + applyLang() 刷新所有静态文案与滑块,并同步切换 TTS 语言。
  • 双镜头分屏(赛车):一个渲染器、两个相机,用 scissor + viewport 渲染同一场景两遍;预警标记用 layers 图层隔离,只在预览镜头出现。

⑨工程教训:反复出现的坑

~25 个审查 agent 在静态/CDP 核查中修复了大量真实硬伤,其中几类反复出现、极具代表性,值得作为浏览器游戏的通用 checklist:

录像为空(0.1KB)MediaRecorder.start() 不带 timeslice,只在 stop 那刻产一块数据;若期间 canvas 被 resize 重置就丢帧start(250) 周期落块 + resize 仅在尺寸真变时重设画布
高速碰撞穿透碰撞判定用固定窗口,满速单帧位移超过窗口记录 prevZ,跨过判定面才触发一次
"新纪录"误报先把 best 更新成本局分,再用 score≥best 判断先捕获 isNew = score>best 再更新 best
AI 演示卡死/冻结AI 取整把当前格当成墙、或某分支永不落子,循环停滞画面静止就近回退到合法格 / 移动失败即兜底动作
过关瞬间竞态过关有 setTimeout 延迟,期间球落底误扣命 / 重复结算加 clearing 过渡锁,期间不扣命不重入

⑩生产方法论:多智能体编排(ultracode)

34 款里的 30 款霓虹游戏由多智能体工作流并行产出。每批的结构是一条流水线:

  • 建造阶段:N 个 agent 并行各做一款,共享同一份"风格 + 录制 + 稳健性"规范(SA + REC),各自带本游戏的 AI 策略说明;
  • 审查阶段:每款再跟一个对抗式审查 agent,通读代码、直接 Edit 修硬伤(因主控无法在浏览器实跑,审查 agent 担任兜底质检,部分还用 Chrome DevTools Protocol 实测)。

共约 50 个 agent、数百万 token 并行产出。这套"建造 + 对抗审查"的流水线把"我没法亲手跑每个游戏"的盲区,转化为可规模化的并行质检。规范本身也在迭代——把前一批踩的坑(如录空)写进下一批的硬性要求,后续批次几乎一次成型。

⑫局限与权衡

  • 无真·联机:单文件离线做不了真实时多人(需信令/服务器);"多人"以本地双人(同屏)实现。
  • 硬件交互不可自动演示:摄像头/语音游戏需真人在场,无法无人值守出预告片。
  • 伪 3D vs 真 3D:赛车/跑酷以伪 3D 换取性能与简洁,牺牲了真实曲面/自由视角。
  • 自写物理 vs 引擎:多数物理(高尔夫/弹球/大西瓜)为自写,可靠但不及 matter.js 真实;引入引擎则增加体积与复杂度。
  • 渲染管线的实时性鸿沟:HyperFrames 渲不了实时游戏,必须"先录后嵌",这是预告片管线的根本约束。

⑬结论

这 34 款游戏共同证明了一件事:浏览器是一个被低估的、完整的游戏平台。从 Canvas 2D 到 WebGL 3D、从 GLSL 着色器到 DOM/SVG、从自写物理到物理引擎、从键鼠触屏到摄像头/语音/陀螺仪——Web 平台原生就能覆盖几乎所有游戏类型,且无需安装、跨端、离线可玩。

而真正的杠杆在于:一套通用底座 + 可替换的"四个旋钮" + 多智能体并行生产。底座保证了 80% 的复用,旋钮决定了类型差异,多智能体编排把"做一款"的成本摊薄到可以"一次做一批"。最终,从一个立方体到 34 款游戏,走的不是 34 条路,而是同一条路走了 34 遍——每遍只换一个旋钮。
